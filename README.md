# 인천대 알림 모음 사이트

## 한 줄 목표

인천대학교 여러 사이트의 공지를 처음 추가할 때 지금까지 올라온 것까지 전부 모으고, 이후로는 새로 올라온 것만 계속 쌓아가면서, 내 학과와 관련 없는 공지는 접어서 보여주는 웹사이트를 만든다. (`auto-update-site` 스킬 기반: GitHub Actions로 매일 수집 → GitHub Pages로 무료 공개)

## 작업 카드

```text
목표: 인천대 공지 사이트를 매일 자동으로 스크래핑해서 최신 글을 모으고, 다른 학과 전용 공지는 걸러서 보여주는 웹사이트를 GitHub Pages로 공개하기
입력: 사용자가 보내줄 공지 사이트 목록 (input/sites.csv), 내 학과 키워드 (input/my_keywords.txt)
출력: docs/index.html (GitHub Pages로 공개되는 실제 웹사이트), data/data.json (수집 데이터)
성공 기준: 매일 아침 8시(KST) GitHub Actions가 자동으로 실행되어 사이트가 최신 공지로 갱신된다
오늘 만든 버전: 로컬 스크립트 완성(collect.py + build_site.py) + GitHub Actions 워크플로 작성. 실제 GitHub 배포는 아래 "GitHub 배포 절차"대로 진행 필요
```

## 구조

```text
incheon-univ-notices/
├── input/
│   ├── sites.csv               ← 수집할 사이트 목록 (사람이 관리)

│   └── everytime_reminders.csv ← 요일마다 바로가기 카드를 만들 사이트 목록 (사람이 관리)

│   └── data.json          ← 수집된 공지 데이터 (collect.py가 매일 갱신)

│   └── build_site.py      ← data.json → docs/index.html 생성
├── docs/
│   └── index.html         ← 실제 공개되는 웹사이트 (GitHub Pages)

```

## 수집 방식

- **사이트를 처음 추가했을 때** (그 사이트 기록이 아직 하나도 없을 때):
  - 최근 3일 안에 올라온 글은 무조건 포함한다.
  - 그보다 오래된 글이라도, 제목에 적힌 신청/행사 기간이 **오늘이거나 오늘 이후까지** 걸쳐 있으면 포함한다. 아직 시작 안 한 공지("7/20부터 접수")도 포함하고, **마감일이 이미 지난 게 확실한 것만** 뺀다.
  - 게시판을 뒤로 넘기다가 어느 페이지에 해당하는 글이 하나도 없으면 그 지점에서 멈춘다. 최대 30페이지까지 확인한다.
  - 최근 며칠 안에 올라온 것 + 지금도(또는 앞으로도) 유효한 것을 실제로 인천대 공지사항에 적용해보니 6페이지, 60건이 나왔다.
- **그 다음부터 업데이트할 때**: 이전에 이미 모아둔 공지는 절대 지우지 않고 그대로 두고, 그 이후 새로 올라온 공지만 찾아서 추가한다. 한 페이지 안에 이미 아는 글이 하나라도 섞여 있으면 그 페이지가 "새 글과 예전 글의 경계"라고 보고 페이지 넘기기를 멈춘다. (업데이트 때는 기간 계산을 다시 하지 않는다 — 새 글인지 아닌지만 본다.)
- 그래서 어느 날 새 글이 하나도 없어도 이전에 모아둔 공지들이 사이트에서 사라지지 않는다.
- 글이 얼마나 오래됐는지는 상관하지 않는다. 제목에 적힌 기간이 지금도 유효하면 몇 달 전 글이라도 포함한다(학과 홈페이지처럼 상단에 고정해두고 학기 내내 걸어두는 공지가 있어서, 날짜로 자르면 오히려 놓치는 게 많았다).
- 사이트당 최대 300건까지 보관한다(계속 쌓이기만 해서 너무 커지는 것 방지). 필요하면 나중에 기간별로 더 다듬을 수 있다.
- 일부 게시판은 "상단 고정" 공지가 모든 페이지 맨 위에 똑같이 반복해서 나온다(데이터과학과 홈페이지가 그렇다). 이런 글이 페이지마다 중복으로 저장되지 않도록, 이미 담은 링크는 건너뛰도록 만들어뒀다.

## 목록 구성

- 사이트별로 섹션을 나누지 않고, 모든 사이트의 공지를 **하나의 목록**으로 합쳐서 보여준다.
- 각 공지 옆에 작은 태그로 **출처**(예: "🏫 데이터과학과 홈페이지 공지사항")를 표시해서, 어느 사이트에서 온 공지인지 바로 알 수 있다.
- 전체 목록은 **날짜 최신순**으로 정렬된다 (사이트 상관없이 가장 최근 글이 맨 위).

## 출처 목록 보기

맨 위 파란 상자(제목이 적힌 곳)에 작은 **"📋 출처"** 버튼이 있다. 누르면 지금 공지를 가져오고 있는 사이트 전체 목록이 펼쳐지고, 각 사이트 이름을 누르면 그 사이트로 바로 이동한다(새 탭으로 열림). 다시 버튼을 누르면 목록이 접힌다. 실제 글을 긁어오는 사이트(`input/sites.csv`)와 에브리타임 바로가기 알림(`input/everytime_reminders.csv`)이 전부 여기 나온다 — 사이트를 추가하거나 빼면 이 목록도 자동으로 같이 바뀐다.

## 에브리타임 바로가기 알림

에브리타임(에타)은 로그인해야만 글을 볼 수 있는 비공개 커뮤니티라서 실제 글은 못 긁어온다. 대신 정해진 요일마다 "이 게시판 확인해보세요"라는 **바로가기 카드**가 다른 공지처럼 하나씩 올라온다.

- `input/everytime_reminders.csv`에 등록된 사이트: 사이트명, 링크, 요일(쉼표로 구분, 월~일)
  - 에타 정보게시판 (월, 목)
  - 에타 홍보게시판 (화, 금)
  - 에타 동아리/학회 (수, 토)
- 해당 요일에 GitHub Actions가 실행되면 "OO 보기"라는 제목의 카드가 새로 하나 생기고, 누르면 그 에브리타임 게시판 주소로 바로 이동한다.
- 다른 공지와 똑같이 읽음/즐겨찾기 버튼이 그대로 적용되고, 탭에도 똑같이 나온다.
- **매번 새 글처럼 다시 올라온다**: 같은 링크를 계속 쓰면 한 번 읽음 표시한 뒤로는 다시 안 뜨기 때문에, 카드마다 그날 날짜를 링크 끝에 몰래 붙여서(`?remind=2026-07-18`) 매번 새로운 글로 인식되게 만들었다. 눌렀을 때 이동하는 주소는 그 부분을 무시하고 항상 원래 게시판으로 정상적으로 열린다.
- 새 사이트를 추가하거나 요일을 바꾸고 싶으면 `input/everytime_reminders.csv`만 고치면 된다 (요일은 "월,목"처럼 쉼표로 구분).

## 탭(폴더) 기능

화면 위쪽에 탭 4개가 있고, 클릭하면 그 탭에 해당하는 공지만 보인다. 각 탭 이름 옆에 개수가 실시간으로 표시된다.

- **안 읽은 공지** (기본으로 보이는 탭): 나와 관련된 공지 중 아직 안 읽은 것. **날짜 최신순**으로 정렬된다.
- **읽은 공지**: 나와 관련된 공지 중 읽음으로 표시한 것. 다른 탭과 달리 **최근에 읽은 순**으로 정렬된다(방금 읽은 게 맨 위, 예전에 읽은 건 아래로). 이미 읽은 걸 또 읽음으로 표시하면(안읽음→다시 읽음) 맨 위로 다시 올라온다.
- **★ 주요 공지**: 별표(즐겨찾기) 표시한 것만 모아서 본다. 읽었든 안 읽었든 상관없이 별표만 하면 여기 나온다. **날짜 최신순**으로 정렬된다.
- **다른 학과 공지**: 학과 필터에 걸린 공지. 읽음/별표 버튼은 없고, 대신 **"관련 공지로 표시"** 버튼만 있다. **날짜 최신순**으로 정렬된다.

버튼 종류:

- **읽음으로 표시 / 읽지 않음으로 표시**: "안 읽은 공지"/"읽은 공지" 탭 사이를 오간다. 공지 제목 링크를 직접 클릭해서 열어도 자동으로 읽음 처리된다. ("다른 학과 공지"에는 이 버튼이 없다 — 요청하신 대로 읽음 추적 대상에서 제외했다.)
- **☆ 주요 공지로 표시 / ★ 주요 공지 해제**: "★ 주요 공지" 탭에 나타나게 하거나 뺀다. 읽음 상태와는 무관하게 독립적으로 동작한다.
- **관련 공지로 표시 / 다른 학과로 되돌리기**: "다른 학과 공지" 탭에 있는 걸 눌러서 표시하면, 그 공지가 "안 읽은 공지" 탭(기본 탭)으로 옮겨진다. 다시 누르면("다른 학과로 되돌리기") 원래대로 "다른 학과 공지" 탭으로 돌아간다.
- **스와이프(밀기)로 읽음 표시**: 공지 카드를 손가락(또는 마우스)으로 왼쪽이나 오른쪽으로 어느 정도 밀면, 카드가 그 방향으로 스르륵 사라지면서 자동으로 "읽음으로 표시" 버튼을 누른 것과 똑같이 처리된다. 살짝만 밀고 놓으면 원래 자리로 되돌아간다(너무 조금 민 건 실수로 처리). 위아래로 더 많이 움직였으면(화면 스크롤 의도) 스와이프로 취급하지 않는다. 버튼 위에서는 스와이프가 시작되지 않아서 버튼은 항상 그대로 눌린다. 제목 글자 위에서 시작해도 스와이프는 정상 동작하고, 진짜로 많이 끌었을 때만 "눌러서 열기"가 취소된다(살짝 탭하면 평소처럼 열림). ("다른 학과 공지"는 애초에 읽음 기능이 없으니 스와이프도 동작하지 않는다.)

**저장 방식(중요)**: 읽음/별표/관련공지 표시는 전부 **이 브라우저에만 저장**된다 (localStorage, 서버·저장소에는 안 남음). 그래서:

- 같은 브라우저로 다시 들어오면 표시가 그대로 유지된다. 매일 사이트가 새로 빌드돼도(GitHub Actions) 안 사라진다.
- 휴대폰·다른 컴퓨터·다른 브라우저에서는 따로 표시해야 한다 (기기 간 동기화 안 됨).
- 브라우저 방문 기록/저장 데이터를 지우면 표시도 같이 사라진다.

## 메모

- **"읽은 공지" 탭만 다르게 정렬되는 이유**: 다른 탭은 전부 공지가 원래 올라온 날짜 순서인데, "읽은 공지" 탭만 "내가 그 공지를 읽음으로 표시한 시점" 기준으로 정렬된다. 서버가 만드는 화면 순서는 그대로 두고, 자바스크립트가 화면에 보여줄 때만(`order` 스타일 값) 살짝 바꿔치기하는 방식이라 데이터 자체는 안 바뀐다.
- **이 정렬이 휴대폰에서 안 되던 문제 (해결 확인됨)**: 컴퓨터 브라우저에서는 잘 되는데 휴대폰에서는 순서가 안 바뀐다는 제보를 받았다. 일부 휴대폰 브라우저가 `order` 값만 슬쩍 바꿨을 때 화면을 바로 다시 안 그리는 경우가 있다고 보고 "강제로 한 번 더 계산하게 만드는" 안전장치(`void document.body.offsetHeight`)를 추가했고, 사용자가 휴대폰에서 실제로 잘 되는 것을 확인해줬다.
- **스와이프가 컴퓨터에서만 되고 휴대폰에서는 안 되던 문제**: 처음에는 "제목 글자(링크) 위에서 스와이프가 시작되면 실수로 링크가 눌릴 수 있다"고 생각해서 제목 위에서는 스와이프를 아예 막아뒀다. 그런데 컴퓨터에서는 마우스로 카드의 빈 여백이나 태그 부분을 끌기 쉬운 반면, 휴대폰에서는 제목 글자가 카드에서 가장 넓은 면적을 차지해서 대부분 그 위에서 밀게 된다 — 그래서 컴퓨터에서는 잘 되고 휴대폰에서는 안 됐다. 지금은 제목 위에서도 스와이프가 시작되도록 바꾸고, 대신 실제로 많이 끌었을 때만 "눌러서 열기"를 취소하는 방식으로 고쳤다.
- **에브리타임은 직접 스크래핑하지 않기로 함**: 로그인해야 글이 보이는 비공개 커뮤니티라서, 로그인 자동화를 하면 이용약관 위반 가능성과 계정 정지 위험이 있고, 로그인 정보를 이 프로젝트(공개 저장소)에 저장하는 것도 안전하지 않다고 판단했다. 대신 실제 글 없이 "이 게시판 확인해보세요"라는 바로가기 카드만 정기적으로 올려주는 방식으로 타협했다(위 "에브리타임 바로가기 알림" 참고).
- **데이터 방식이 "증분 누적"으로 바뀜**: 처음에는 "오늘 날짜인 글만" 매번 다시 계산하는 방식이었는데, 이러면 어느 날 새 글이 없으면 사이트가 통째로 비어 보이는 문제가 있었다. 그래서 "이미 모은 건 계속 보관 + 새로 나온 것만 추가"하는 증분 방식으로 바꿨다.
- **처음 추가한 사이트를 다시 채우려면**: 이미 몇 건이 저장된 상태에서 "더 옛날 것까지 가져와줘"라고 요청해도, 증분 수집은 이미 아는 글을 만나면 바로 멈추기 때문에 자동으로는 더 못 가져온다. 이럴 땐 그 사이트의 저장된 기록을 한 번 비우고 다시 수집(재부트스트랩)해야 한다 — Codex에게 요청하면 된다.
- **제목의 신청/행사 기간 인식은 완벽하지 않다**: "~7/24", "7월 20일부터 7월 31일까지", "7.20~7.31", "7/20(월) ~ 8/21(금)"(요일 표시 포함) 같은 흔한 표현은 인식하지만, 사람이 쓰는 표현은 워낙 다양해서 못 알아채는 경우도 있을 수 있다. 못 알아챈 글도 최근 3일 안에 올라온 거면 어차피 포함되고, 놓친 게 있어 보이면 알려주면 규칙을 더 추가할 수 있다.
- **다른 게시판 템플릿도 지원**: 인천대 사이트 중에는 제목 링크가 진짜 주소(href) 대신 자바스크립트 클릭 이벤트로 되어 있는 곳도 있다(데이터과학과 홈페이지가 그렇다). 그런 사이트는 `parser` 칸에 `inu_js_link`를 적으면 된다 (data 속성에서 글 주소를 직접 조합해서 만든다). 새 사이트를 추가할 때 기존 `inu_standard`로 안 되면 이 방식도 시도해볼 수 있다.
- **"오늘 이후" 공지도 포함**: 처음에는 "지금 이 순간 유효한 기간(시작~마감 사이)"만 포함했는데, 그러면 "다음 주부터 접수 시작"처럼 아직 시작 전인 공지를 놓쳤다. 그래서 마감일이 이미 지난 게 확실한 것만 빼고, 나머지(진행 중이거나 아직 시작 전이거나 마감일이 안 적힌 것)는 다 포함하도록 넓혔다.
- **몇몇 사이트는 시작점을 다르게 채움**: 데이터과학과 홈페이지는 일반 규칙(최근 3일 + 기간이 유효한 것)으로는 2건만 나왔는데, "처음엔 1~3페이지 전부를 필터 없이 채워달라"는 요청으로 39건을 통째로 채웠다. 데이터과학과 장학게시판·학산도서관 공지사항·인천대 기숙사 공지사항은 "N페이지 모든 공지를 필터 없이 채워달라"는 요청으로 각각 11건, 10건, 10건을 채웠다. 전부 자동화된 규칙이 아니라 그때 한 번 스크립트를 따로 돌려서 채운 것이라, 나중에 사이트를 또 하나 추가할 때 이 방식이 자동으로 적용되진 않는다 — 필요하면 다시 요청하면 된다. 시작점을 채운 이후부터는 모든 사이트가 똑같이 "새로 올라온 글만, 날짜 상관없이" 증분 수집된다.
- **기숙사 공지사항도 자바스크립트 링크 템플릿**: `dorm.inu.ac.kr`은 데이터과학과 홈페이지와 똑같이 제목 링크가 진짜 주소 대신 `data-site-id`/`data-fnct-no`/`data-bbs-artcl-seq` 속성으로 되어 있다. 처음에 `inu_standard` 파서로 테스트했더니 5건 전부 링크가 게시판 목록 주소 하나로 똑같이 나오는 버그가 있어서(빈 href를 그대로 조합), `inu_js_link` 파서로 바꿔서 정상적인 개별 글 주소를 확인한 뒤 등록했다. 새 인천대 사이트를 추가할 때는 이렇게 "링크가 전부 똑같이 나오면" JS 링크 템플릿일 가능성이 크다는 걸 기억해두면 된다.
- **1페이지만 채워둔 사이트가 옛날 글을 통째로 긁어오던 버그 (발견 및 수정)**: 기숙사 사이트를 추가하고 하루 지나서 자동 업데이트를 확인하던 중, 학산도서관 공지사항이 갑자기 10건에서 150건(안전 상한)까지 늘어나는 걸 발견했다. 원인은 증분 수집이 "한 페이지가 통째로 새 글일 때만" 다음 페이지로 넘어가는 방식이었는데, 학산도서관처럼 맨 처음 1페이지만 시작점으로 채워둔 사이트는 2페이지부터가 "한 번도 저장한 적 없는 영역"이라서 이미 아는 글이 하나도 없다 — 그래서 계속 "새 글처럼" 보여서 안전 상한(15페이지)까지 몇 달 전 글까지 다 긁어올 뻔했다. "한 페이지 안에 이미 아는 글이 하나라도 섞여 있으면 그 페이지에서 멈춘다"로 조건을 바꿔서 고쳤다. 데이터과학과 장학게시판·인천대 기숙사 공지사항도 똑같이 1페이지만 채워둔 사이트라 같은 위험이 있었는데 이번 수정으로 같이 해결됐다.
- **인천대 기숙사 공지사항을 결국 제거함 (GitHub Actions에서 접속 차단)**: cron-job.org로 예약 실행을 안정화한 뒤에도, 기숙사 사이트(`dorm.inu.ac.kr`)만 GitHub Actions 실행 때마다 계속 연결 타임아웃이 났다. 로컬 컴퓨터에서는 항상 바로 접속되는데, 서로 다른 실행(=서로 다른 GitHub 서버 IP)에서도 매번 똑같이 실패하는 걸 보고, 학교 쪽에서 GitHub Actions가 쓰는 클라우드 IP 대역(Azure) 자체를 막고 있는 것으로 판단했다. 코드로 확실히 뚫을 방법이 없고 기숙사 게시판은 자주 올라오는 곳도 아니라서, 사용자와 상의 후 `input/sites.csv`와 `data/data.json`에서 완전히 제거했다(출처 목록도 7개로 원복).
- **GitHub Actions 예약 실행이 며칠씩 통째로 안 돈 적이 있었다 (외부 스케줄러로 해결)**: 처음엔 정각(`0 23 * * *`)이 GitHub 혼잡 시간대라 그런 줄 알고 08:07 KST(`7 23 * * *`)로 옮겼는데, 그렇게 해도 그다음 이틀(7/20, 7/21) 다 자동 실행이 안 됐다 — 저장소가 생긴 이후 `schedule` 트리거로 성공한 건 단 1번뿐이었다. 즉 "정각 혼잡"이 진짜 원인이 아니라, GitHub의 예약 실행 자체가 개인 계정 저장소에서 신뢰도가 낮은 게 근본 원인이었다. **cron-job.org**(무료 외부 스케줄러)가 매일 08:07 KST에 GitHub API로 `workflow_dispatch`(수동 실행과 동일한 신호)를 대신 호출하도록 설정해서 해결했다. GitHub 네이티브 `schedule` 트리거는 그대로 남겨뒀다(어쩌다 도는 날은 보너스, `concurrency` 설정 덕분에 겹쳐도 안전). 자세한 설정 절차는 `.agents/skills/auto-update-site/references/github-setup.md`의 "6. 외부 스케줄러로 예약 실행 안정화" 참고.
- **자바스크립트로만 그려지는 사이트(학산도서관)는 다른 방식으로 접근**: 이 사이트는 페이지 소스를 그냥 받아오면(`requests`) 내용이 텅 빈 껍데기만 나온다(화면을 자바스크립트/Angular가 그리기 때문). 이럴 땐 그 화면이 "뒤에서 어떤 주소로 데이터를 요청해서 받아오는지"를 찾아야 한다. 이번엔 실제로 브라우저를 잠깐 띄워서(Playwright) 화면이 어떤 요청을 보내는지 엿본 다음, 그 주소(JSON API, `pyxis-api/1/bulletin-boards/1/bulletins`)를 알아내서 `requests`로 그 주소를 직접 불러오는 방식으로 만들었다. 한 번 이렇게 찾아두면 이후 실제 자동 수집(GitHub Actions)에서는 브라우저 없이 `requests`만으로 매일 가볍게 동작한다.
- **인천대 홈페이지의 "카테고리별 게시판"은 사실 같은 글을 공유한다**: `www.inu.ac.kr`에는 "인천대 홈페이지 공지사항"(fnctNo=2006, 전체) 말고도 "학사공지"(246), "일반/행사/모집"(2611)처럼 카테고리별로 보이는 화면이 여러 개 있는데, 실제로 확인해보니 이건 서로 다른 공지가 아니라 **같은 글 저장소를 카테고리로 걸러서 보여주는 화면**이었다(같은 글 번호가 `/2006/427133/...`와 `/2611/427133/...` 양쪽 주소로 다 열림). 그래서 처음엔 "인천대 학사공지"(246)를 별도 사이트로 추가했다가, 나중에 "일반/행사/모집"(2611)을 추가하려던 중 이 사실을 발견하고 학사공지(246)를 삭제해서 전체(2006)로 통합했다(중복 없이 40건을 옮김). **앞으로 인천대(www.inu.ac.kr) 사이트를 추가할 때는, 그 게시판이 "전체(2006)"의 카테고리 필터인지부터 먼저 확인해야 한다** — 같은 글 번호(artclSeq)가 다른 fnctNo 주소로도 열리는지 확인해보면 알 수 있다.
- **AI 분석 미사용**: `auto-update-site` 템플릿은 기본적으로 Gemini API로 데이터를 분석하지만, 이 프로젝트는 학과 이름 패턴 매칭만으로 충분해서 AI 호출 없이 만들었다. API 키/Secret 등록이 필요 없다.
- **필터링 한계**: "HUSS포용사회이니셔티브학부"처럼 이름 끝이 "학부"로 끝나지만 실제로는 전교생 대상인 공지가 다른 학과로 잘못 접힐 수 있다. 걸러진 공지도 숨기지 않고 펼쳐보기로 항상 확인 가능하게 해뒀다.
- **저장소 분리 이유**: 메인 워크스페이스(BAI-GUIDE) 저장소는 개인 메모 등 비공개 성격 파일이 섞여 있어서, 공개 웹사이트로 배포되는 이 프로젝트는 별도의 전용 저장소로 분리하기로 했다 (사용자 확인 완료). 메인 저장소의 `.gitignore`에 이 폴더가 추가되어 있다.
