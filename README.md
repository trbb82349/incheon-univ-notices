# 인천대 알림 모음 사이트

## 한 줄 목표

인천대학교 여러 사이트의 공지를 처음 추가할 때 지금까지 올라온 것까지 전부 모으고, 이후로는 새로 올라온 것만 계속 쌓아가면서, 내 학과와 관련 없는 공지는 접어서 보여주는 웹사이트를 만든다. (`auto-update-site` 스킬 기반: GitHub Actions로 매일 수집 → GitHub Pages로 무료 공개)

## 작업 카드

```text
목표: 인천대 공지 사이트를 매일 자동으로 스크래핑해서 최신 글을 모으고, 다른 학과 전용 공지는 걸러서 보여주는 웹사이트를 GitHub Pages로 공개하기
입력: 사용자가 보내줄 공지 사이트 목록 (input/sites.csv), 내 학과 키워드 (input/my_keywords.txt)
출력: docs/index.html (GitHub Pages로 공개되는 실제 웹사이트), data/data.json (수집 데이터)
성공 기준: 매일 아침 8시(KST) GitHub Actions가 자동으로 실행되어 사이트가 최신 공지로 갱신된다
오늘 만든 버전: 로컬 스크립트 완성(collect.py + build_site.py) + GitHub Actions 워크플로 작성. 실제 GitHub 배포는 아래 "GitHub 배포 절차"대로 진행 필요
```

## 구조

```text
incheon-univ-notices/
├── input/
│   ├── sites.csv          ← 수집할 사이트 목록 (사람이 관리)
│   └── my_keywords.txt    ← 내 학과 키워드 (사람이 관리)
├── data/
│   └── data.json          ← 수집된 공지 데이터 (collect.py가 매일 갱신)
├── src/
│   ├── collect.py         ← 사이트 스크래핑 + 학과 필터링 → data.json 저장
│   └── build_site.py      ← data.json → docs/index.html 생성
├── docs/
│   └── index.html         ← 실제 공개되는 웹사이트 (GitHub Pages)
└── .github/workflows/
    └── update.yml         ← 매일 08:00 KST 자동 실행
```

- [x] **GitHub 저장소 생성 + 첫 배포 완료** (`trbb82349/incheon-univ-notices`, Pages 켜짐, Actions 정상 동작 확인)

## 수집 방식

- **사이트를 처음 추가했을 때** (그 사이트 기록이 아직 하나도 없을 때):
  - 최근 3일 안에 올라온 글은 무조건 포함한다.
  - 그보다 오래된 글이라도, 제목에 "~7/24", "7월 20일부터 7월 31일까지" 같은 신청/행사 기간이 적혀 있고 그 기간에 오늘 날짜가 들어가면 포함한다 (예: 7/13에 올라온 글이 "~7/24까지"라고 되어 있으면, 오늘이 7/18이니까 아직 기간 안이라 포함).
  - 게시판을 뒤로 넘기다가 어느 페이지에 해당하는 글이 하나도 없으면 그 지점에서 멈춘다. 최대 30페이지까지 확인한다.
  - 최근 며칠 안에 올라온 것 + 지금도 기간이 유효한 것을 실제로 인천대 공지사항에 적용해보니 6페이지, 60건이 나왔다.
- **그 다음부터 업데이트할 때**: 이전에 이미 모아둔 공지는 절대 지우지 않고 그대로 두고, 그 이후 새로 올라온 공지만 찾아서 추가한다. 이미 알고 있는 글(링크로 구분)을 만나면 그 뒤로는 다 예전 글이라는 뜻이므로 페이지 넘기기를 멈춘다. (업데이트 때는 기간 계산을 다시 하지 않는다 — 새 글인지 아닌지만 본다.)
- 그래서 어느 날 새 글이 하나도 없어도 이전에 모아둔 공지들이 사이트에서 사라지지 않는다.
- 30일보다 오래된 글은 기간이 적혀 있어도 더 이상 확인하지 않는다 (오탐 방지 + 무한정 뒤로 가는 것 방지).
- 사이트당 최대 300건까지 보관한다(계속 쌓이기만 해서 너무 커지는 것 방지). 필요하면 나중에 기간별로 더 다듬을 수 있다.

## 메모

- **데이터 방식이 "증분 누적"으로 바뀜**: 처음에는 "오늘 날짜인 글만" 매번 다시 계산하는 방식이었는데, 이러면 어느 날 새 글이 없으면 사이트가 통째로 비어 보이는 문제가 있었다. 그래서 "이미 모은 건 계속 보관 + 새로 나온 것만 추가"하는 증분 방식으로 바꿨다.
- **처음 추가한 사이트를 다시 채우려면**: 이미 몇 건이 저장된 상태에서 "더 옛날 것까지 가져와줘"라고 요청해도, 증분 수집은 이미 아는 글을 만나면 바로 멈추기 때문에 자동으로는 더 못 가져온다. 이럴 땐 그 사이트의 저장된 기록을 한 번 비우고 다시 수집(재부트스트랩)해야 한다 — Codex에게 요청하면 된다.
- **제목의 신청/행사 기간 인식은 완벽하지 않다**: "~7/24", "7월 20일부터 7월 31일까지", "7.20~7.31" 같은 흔한 표현은 인식하지만, 사람이 쓰는 표현은 워낙 다양해서 못 알아채는 경우도 있을 수 있다. 못 알아챈 글도 최근 3일 안에 올라온 거면 어차피 포함되고, 놓친 게 있어 보이면 알려주면 규칙을 더 추가할 수 있다.
- **AI 분석 미사용**: `auto-update-site` 템플릿은 기본적으로 Gemini API로 데이터를 분석하지만, 이 프로젝트는 학과 이름 패턴 매칭만으로 충분해서 AI 호출 없이 만들었다. API 키/Secret 등록이 필요 없다.
- **필터링 한계**: "HUSS포용사회이니셔티브학부"처럼 이름 끝이 "학부"로 끝나지만 실제로는 전교생 대상인 공지가 다른 학과로 잘못 접힐 수 있다. 걸러진 공지도 숨기지 않고 펼쳐보기로 항상 확인 가능하게 해뒀다.
- **저장소 분리 이유**: 메인 워크스페이스(BAI-GUIDE) 저장소는 개인 메모 등 비공개 성격 파일이 섞여 있어서, 공개 웹사이트로 배포되는 이 프로젝트는 별도의 전용 저장소로 분리하기로 했다 (사용자 확인 완료). 메인 저장소의 `.gitignore`에 이 폴더가 추가되어 있다.
- 학교 사이트에 과도하게 자주 요청을 보내지 않도록 사이트별 요청 사이에 1초 딜레이를 넣어뒀다.
