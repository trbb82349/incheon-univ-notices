# 사용된 컴퓨터 지식/스킬 — 인천대 알림 모음

## 프로젝트 요약

인천대학교 여러 사이트의 공지를 자동으로 모아 보여주는 웹페이지를 만드는 중이다. 인천대 홈페이지 공지사항 게시판 1곳을 실제로 연동했고, 내 학과와 관련 없는 다른 학과 전용 공지를 자동으로 걸러서 접어두는 기능도 만들었다. `auto-update-site` 스킬을 적용해서 매일 아침 8시에 GitHub이 알아서 공지를 다시 확인하고 웹사이트를 갱신하도록 구조를 바꿨고(collect.py로 수집 → build_site.py로 웹페이지 생성 → GitHub Actions가 매일 자동 실행 → GitHub Pages로 공개), 실제로 GitHub 저장소(`incheon-univ-notices`)를 만들어 배포까지 완료했다. 이후 수집 방식을 "그날 날짜만 매번 새로 계산"하던 것에서 "사이트를 처음 추가할 때 최근 글 + 제목에 적힌 신청기간이 아직 유효한 글까지 다 모으고, 그 다음부터는 새로 올라온 것만 추가로 쌓는" 방식으로 바꿨다(예전 글이 사라지는 문제, 그리고 며칠 전에 올라왔지만 마감이 아직 안 지난 공지를 놓치는 문제를 둘 다 해결). 공지마다 "읽음으로 표시" 버튼을 처음 달았다가, 이후 사용자 요청으로 "안 읽은 공지/읽은 공지/★ 주요 공지/다른 학과 공지" 4개 탭으로 화면을 나누고, 별표(즐겨찾기) 기능과 "다른 학과 공지를 관련 공지로 재분류하는" 기능까지 추가했다. "휴대폰에서 주로 볼 거라 모바일에 맞춰 귀엽고 하늘색 테마로 꾸며달라"는 요청을 받아 전체 디자인을 모바일 우선 + 하늘색 파스텔 톤으로 새로 꾸몄고, 색이 쨍하다는 피드백을 받아 채도를 낮췄다. 이후 데이터과학과 홈페이지 공지사항을 두 번째 사이트로 추가하면서, 그 사이트의 게시판 구조가 미묘하게 달라서(제목 링크가 진짜 주소 대신 자바스크립트로 되어 있음, 상단 고정 글이 모든 페이지에 반복됨) 새 파서와 중복 방지 로직을 추가했고, 그 과정에서 "글이 오래되면 기간이 적혀 있어도 무시하던" 버그와 "날짜 사이에 요일 표시가 있으면 기간을 잘못 읽던" 버그를 발견해서 고쳤다. 이후 "아직 시작 안 한(오늘 이후에 시작하는) 공지도 포함해달라"는 요청을 받아, "지금 이 순간 기간 안"인 것만 포함하던 조건을 "마감일이 이미 지난 것만 빼고 나머지는 다 포함"하는 조건으로 넓혔다. 그래도 데이터과학과 사이트는 2건만 나오자, "처음엔 그냥 1~3페이지 전부를 필터 없이 채워달라"는 요청을 받아 그 사이트만 예외로 39건을 통째로 채워 넣었다(일반 규칙을 바꾼 게 아니라 이번 한 번 수동으로 채운 것). 사이트가 2개로 늘어나면서 화면에 "인천대 홈페이지"/"데이터과학과 홈페이지" 섹션이 따로 나뉘어 보였는데, "섹션 구분 없이 하나로 합치고 대신 출처 태그를 붙이고 최신순으로 정렬해달라"는 요청을 받아 화면 구조를 다시 바꿨다. 이어서 데이터과학과 장학게시판(같은 사이트의 다른 게시판)을 세 번째로 추가했는데, 같은 사이트라 기존에 만들어둔 `inu_js_link` 파서를 코드 수정 없이 그대로 재사용할 수 있었다. 네 번째로 학산도서관 공지사항을 추가할 땐, 그 사이트가 지금까지와 완전히 다른 방식(화면 전체를 자바스크립트로 그리는 Angular 앱)으로 만들어져 있어서, 브라우저를 잠깐 띄워 실제 데이터가 어디서 오는지 엿본 뒤 그 뒤에 숨은 JSON API를 직접 호출하는 새로운 방식의 파서를 만들었다. 다섯 번째 인천대 학사공지는 다시 첫 번째 사이트와 같은 표준 템플릿이라 새 코드 없이 바로 추가됐다(1~4페이지 40건을 필터 없이 채움). 그런데 여섯 번째로 "일반/행사/모집"(2611) 사이트를 추가하려다가, 이 게시판이 사실 "인천대 홈페이지 공지사항"(2006, 전체)과 같은 글을 카테고리로만 걸러서 보여주는 화면이라는 걸 발견했다(글 번호가 두 주소 모두에서 똑같이 열림). 같은 이유로 이미 추가했던 "인천대 학사공지"(246)도 겹치는 게시판이라는 게 확인돼서, 2611은 아예 추가하지 않고 246은 삭제한 뒤 그 40건을 전체(2006) 사이트로 옮겨 합쳤다. 그다음 사용자가 에브리타임(인천대 커뮤니티 앱)의 정보게시판을 추가해달라고 했는데, 확인해보니 로그인 없이는 글을 볼 수 없는 비공개 커뮤니티였다. 로그인 계정을 넘겨받아 자동화하는 것도 제안받았지만, 이용약관 위반·계정 정지·비밀번호 보관 위험을 이유로 정중히 거절했다. 대신 실제 글을 긁어오지 않고 "이 게시판 보러 가기"라는 바로가기 카드만 요일마다(게시판 3개, 각각 다른 요일 조합) 자동으로 하나씩 올라오도록 만들어서 타협했다. 마지막으로 "공지 카드를 좌우로 밀면(스와이프) 읽음으로 표시되게 해달라"는 요청을 받아, 손가락(또는 마우스)으로 카드를 끄는 동작을 인식해서 충분히 밀면 읽음 처리되고 살짝만 밀면 원래 자리로 돌아오는 기능을 추가했다. 그런데 사용자가 "컴퓨터에서는 되는데 휴대폰에서는 안 된다"고 알려줘서, 제목 글자(가장 넓은 터치 영역) 위에서 스와이프 시작을 막아뒀던 게 원인이라는 걸 찾아 고쳤다. 마지막으로 "다른 탭은 최신순인데 읽은 공지 탭만 최근에 읽은 순으로 정렬해달라"는 요청을 받아, 서버가 만드는 화면 순서는 그대로 두고 "읽은 공지" 탭을 볼 때만 화면 표시 순서를 바꿔치기하는 기능을 추가했다. 그런데 이번에도 "컴퓨터에서는 되는데 휴대폰에서는 순서가 안 바뀐다"는 제보를 받았다. 코드 자체는 컴퓨터/휴대폰을 다르게 처리하지 않아서 확실한 원인을 콕 집어내진 못했지만, 일부 휴대폰 브라우저가 스타일 값만 바꾸면 화면을 바로 안 다시 그리는 경우가 있다고 보고 "강제로 다시 그리게 만드는" 안전장치를 추가했다(실제 기기로 재현 확인은 못 함, 사용자 확인 대기 중).

## 사용한 지식/스킬

- CSV 파일 (엑셀처럼 표 모양 데이터를 저장하는 아주 단순한 텍스트 파일):
  - 어디서 썼는지: `input/sites.csv`에 "사이트 이름, 주소, 분류"를 한 줄씩 적어두는 용도로 썼다.
  - 어떤 효과가 있었는지: 사이트 목록을 코드 안에 직접 적지 않고 파일로 따로 관리하니, 나중에 사이트를 추가하거나 뺄 때 코드를 건드리지 않고 이 파일만 고치면 된다.
- 웹 스크래핑 (사람이 웹사이트를 눈으로 보고 정보를 옮겨 적는 대신, 프로그램이 자동으로 웹페이지 내용을 읽어오는 기술) - requests + BeautifulSoup:
  - 어디서 썼는지: `src/collect.py`의 `scrape_inu_standard` 함수. `requests`로 게시판 페이지를 통째로 받아오고, `BeautifulSoup`로 그 안에서 "제목이 들어있는 자리", "작성 부서가 들어있는 자리" 같은 특정 부분만 골라냈다.
  - 어떤 효과가 있었는지: 사람이 매번 게시판에 들어가서 눈으로 확인할 필요 없이, 스크립트 실행 한 번으로 최신 공지 10개를 자동으로 가져올 수 있게 됐다.
  - 사이트 구조 확인 방법: 브라우저 개발자 도구 대신, `requests`로 페이지를 직접 받아서 파일로 저장한 뒤 그 안에서 반복되는 태그 패턴(여기서는 `<table class="board-table">` 안의 `<tr>` 하나가 공지 1개)을 찾았다.
- 정규표현식(regex)으로 패턴 찾기:
  - 어디서 썼는지: 공지 제목이나 작성 부서 이름에서 "OO학과", "OO학부", "OO전공"처럼 학과 이름 패턴을 자동으로 찾아내는 데 썼다 (`DEPT_PATTERN`).
  - 어떤 효과가 있었는지: "이 공지가 어느 학과 것인지"를 사람이 하나하나 읽지 않고, 이름 끝 글자 패턴만으로 자동 판단할 수 있게 됐다. 다만 완벽하지 않아서(예: 학과 이름이 아닌데 우연히 "학부"로 끝나는 조직명), 걸러진 공지도 숨기지 않고 펼쳐서 확인할 수 있게 안전장치를 뒀다.
- HTML의 `<details>`/`<summary>` 태그로 접이식 목록 만들기 (초기 버전에서 사용, 이후 탭 방식으로 교체됨):
  - 어디서 썼는지: 처음에는 "다른 학과 공지"를 기본은 접혀 있지만 클릭하면 펼쳐지는 형태로 보여주는 데 썼다. 나중에 "탭으로 화면을 나눠달라"는 요청을 받고 나서, 탭 버튼 + 자바스크립트로 보이고 숨기는 방식으로 바꾸면서 이 태그는 더 안 쓰게 됐다.
  - 어떤 효과가 있었는지: 자바스크립트 코드를 한 줄도 안 쓰고, HTML 태그만으로 클릭하면 펼쳐지는 목록을 만들 수 있었다 — 다만 "여러 개의 서로 겹치는 분류(읽음/안읽음/즐겨찾기 등)"를 나타내기엔 한계가 있어서, 요구사항이 복잡해지자 자바스크립트 탭 방식으로 넘어갔다.
- 데이터 수집과 화면 생성을 분리하기 (`data/data.json`을 가운데 두고 `collect.py` / `build_site.py`로 나눔):
  - 어디서 썼는지: `collect.py`는 "사이트에서 정보를 가져와서 파일(json)에 저장"만 담당하고, `build_site.py`는 "그 파일을 읽어서 웹페이지를 만드는" 역할만 담당하도록 나눴다.
  - 어떤 효과가 있었는지: 웹페이지 디자인만 바꾸고 싶을 때 다시 사이트를 스크래핑할 필요 없이 `build_site.py`만 다시 실행하면 된다. 두 역할이 섞여 있으면 나중에 하나를 고칠 때 다른 하나까지 실수로 건드릴 위험이 있다.
- GitHub Actions (사람이 컴퓨터를 켜두지 않아도, GitHub이 정해진 시간에 대신 코드를 실행해주는 무료 서비스):
  - 어디서 썼는지: `.github/workflows/update.yml`에 "매일 08:00 KST(한국시간)에 `collect.py`와 `build_site.py`를 순서대로 실행하고, 바뀐 내용이 있으면 자동으로 커밋·푸시"하도록 설정했다.
  - 어떤 효과가 있었는지: 내 컴퓨터를 매일 켜서 스크립트를 실행할 필요 없이, 자고 있어도 GitHub 서버가 대신 실행해준다.
  - cron 표기법 주의점: GitHub Actions의 cron 시간은 항상 UTC(세계 표준시) 기준이라서, 한국시간 08:00에 실행하려면 `0 23 * * *`(전날 23:00 UTC)처럼 9시간을 빼서 계산해야 한다.
- GitHub Pages (저장소 안의 `docs/` 폴더 내용을 무료로 인터넷에 공개해주는 서비스):
  - 어디서 썼는지: `docs/index.html`이 실제로 사람들이 접속해서 보는 주소(`https://아이디.github.io/저장소명`)가 된다.
  - 어떤 효과가 있었는지: 별도로 서버를 빌리거나 돈을 내지 않아도 웹사이트를 인터넷에 공개할 수 있다. 단, `docs/.nojekyll`이라는 빈 파일을 꼭 넣어야 정상 작동한다(안 넣으면 404 오류가 날 수 있음).
- 게시판 페이지 번호 자동으로 넘기기 (URL의 `?page=1`, `?page=2` 같은 부분을 코드로 바꿔가며 여러 페이지를 순서대로 확인):
  - 어디서 썼는지: `collect.py`의 `with_page` 함수가 주소의 페이지 번호만 바꿔서 다음 페이지 주소를 만들고, `fetch_new_notices` 함수가 필요한 만큼 계속 다음 페이지로 넘어가도록 했다.
  - 어떤 효과가 있었는지: 처음에는 게시판 1페이지(최신 10개)만 가져왔는데, 하루에 공지가 10개 넘게 올라오면 뒷부분을 놓치는 문제가 있었다. 페이지를 자동으로 넘기게 바꾸면서 "그 시점에 올라온 글은 몇 개든 다 가져오기"가 가능해졌다.
- "이미 아는 글" 기준으로 새 글만 골라내기 (링크 주소를 글의 고유 번호표처럼 사용):
  - 어디서 썼는지: `collect.py`가 이전에 저장해둔 공지들의 링크 목록(`known_links`)을 기억해뒀다가, 새로 가져온 글의 링크가 그 목록에 이미 있으면 "이미 아는 글"로 보고 그 지점에서 페이지 넘기기를 멈춘다.
  - 어떤 효과가 있었는지: "오늘 날짜냐 아니냐"로 판단하지 않고 "이미 봤냐 안 봤냐"로 판단하니까, 게시판에 며칠 동안 새 글이 없다가 갑자기 여러 개 올라와도 놓치지 않고 다 잡아낸다. 그리고 이전에 모아둔 글은 지우지 않고 계속 쌓아두기 때문에, 어느 날 새 글이 없다고 사이트가 텅 비어 보이는 일도 없어졌다.
  - 딱 한 가지 예외(사이트를 맨 처음 추가하는 순간)는 비교할 "이미 아는 글"이 하나도 없으므로, 그때는 아래에 나오는 "제목에서 기간 읽어내기"까지 같이 써서 시작점을 만든다. (무한정 넘기지 않도록 최대 15페이지라는 안전장치도 넣었다.)

  - 어디서 썼는지: `playwright`(자바스크립트를 진짜로 실행할 수 있는, 화면 없는 가짜 브라우저)를 임시로 설치해서 그 페이지를 실제로 열어보고, 그동안 브라우저가 인터넷에 어떤 요청을 보내는지 전부 엿들었다(`page.on('response', ...)`). 그 결과 `pyxis-api/1/bulletin-boards/1/bulletins`라는 주소로 깔끔한 JSON(글 제목, 작성자, 날짜가 이미 정리된 데이터)을 받아오고 있다는 걸 알아냈다.
  - 어떤 효과가 있었는지: 한 번 이 주소를 찾아내고 나면, 이후 실제 자동 수집 코드(`scrape_lib_pyxis`)는 `playwright`나 무거운 가짜 브라우저 없이 `requests`만으로 그 JSON 주소를 직접 불러오면 된다 — 사람이 브라우저로 보는 화면과 그 화면이 실제로 받아오는 데이터는 별개라는 걸 이용한 것이다. `playwright`는 "찾을 때 한 번" 쓰고, 매일 자동으로 도는 GitHub Actions에는 설치되지 않는다(가볍게 유지).
  - 밑에 깔린 원리: 페이지 안의 자바스크립트 코드(`main.js`)에는 `"[API_URL]/[HOMEPAGE_ID]/..."`처럼 실제 값이 아닌 자리표시자만 적혀 있고, 앱이 실행되면서 진짜 서버 주소로 바꿔치기하는 구조였다. 이런 코드는 압축된 파일을 눈으로 읽어서 진짜 주소를 알아내기 매우 어려워서(값이 코드에 그대로 안 적혀 있으므로), 코드를 읽는 대신 "실행됐을 때 실제로 어디로 요청이 가는지" 관찰하는 방법이 훨씬 빠르고 확실했다.
- 새 게시판이 "진짜 새로운 글 모음"인지, 아니면 "이미 있는 글을 카테고리로만 걸러 보여주는 화면"인지 확인하기 (글 번호로 교차 확인):
  - 왜 필요했는지: 인천대 홈페이지는 "공지사항"(전체), "학사공지", "일반/행사/모집"처럼 메뉴가 여러 개로 나뉘어 있어서 서로 다른 게시판처럼 보였다. 그런데 실제로는 몇몇 메뉴가 "전체" 게시판의 카테고리 필터일 뿐, 글 자체는 하나의 저장소를 공유하고 있었다.
  - 어디서 썼는지: 새 게시판에서 글 하나의 주소(예: `/bbs/inu/2611/427133/artclView.do`)를 가져와서, 그 글 번호(`427133`)만 그대로 두고 게시판 번호(`fnctNo`)만 이미 등록된 다른 게시판 걸로 바꿔서(`/bbs/inu/2006/427133/artclView.do`) 다시 요청해봤다. 둘 다 똑같은 글이 열리면 "같은 글 저장소를 공유하는 카테고리 필터"라는 뜻이다.
  - 어떤 효과가 있었는지: 이 확인을 안 했으면 같은 공지가 "인천대 홈페이지 공지사항"과 "일반/행사/모집" 두 곳에 중복으로 나타났을 것이다. 새 사이트를 추가하기 전에 "혹시 이미 등록한 사이트랑 같은 글을 공유하지는 않는지" 한 번 의심하고 확인하는 습관이 중요하다는 걸 배웠다.
- 로그인이 필요해서 못 긁어오는 사이트를 위한 "바로가기 알림 카드" 만들기 (링크 끝에 날짜를 몰래 붙이는 트릭):
  - 왜 필요했는지: 에브리타임처럼 실제 글은 못 가져오는 곳도, "이 게시판 확인해보세요"라는 알림만큼은 다른 공지처럼 요일마다 정기적으로 띄우고 싶다는 요청을 받았다. 그런데 이 시스템은 "링크(주소)"를 그 글의 고유 번호표로도 쓰고 있어서, 매번 똑같은 게시판 주소를 링크로 쓰면 처음 한 번 읽음 표시한 뒤로는 "이미 아는 글"로 취급되어 다시는 새 글처럼 안 올라온다.
  - 어디서 썼는지: `collect.py`의 `build_reminder_notice` 함수가 실제 게시판 주소 뒤에 `?remind=2026-07-18`처럼 그날 날짜를 붙여서 매번 "새로운 주소"처럼 만든다. 인터넷 주소에서 물음표 뒤에 붙는 부분(쿼리 파라미터)은 웹사이트가 모르는 값이면 그냥 무시하는 경우가 많아서, 실제로 눌러봐도 정상적으로 원래 게시판이 열린다(직접 요청해서 확인함).
  - 어떤 효과가 있었는지: 기존에 만들어둔 읽음/즐겨찾기/탭 기능을 하나도 안 건드리고도, "매일 새 글처럼 다시 나타나는 알림"을 자연스럽게 만들 수 있었다. "이 시스템에서 무엇이 고유 식별자로 쓰이는지"를 이해하면, 그 식별자만 살짝 바꿔서 원하는 동작(반복 알림)을 만들어낼 수 있다는 걸 배웠다.
- 손가락/마우스로 화면을 밀어서(스와이프) 뭔가를 실행시키기 (Pointer Events):
  - 왜 필요했는지: "카드를 좌우로 밀면 읽음으로 표시되게 해달라"는 요청을 받았다. 터치(손가락)와 마우스는 원래 서로 다른 이벤트(touchstart/mousedown 등)를 쓰는데, 이 둘을 따로 처리하면 코드가 두 배로 늘어난다.
  - 어디서 썼는지: `build_site.py`의 자바스크립트가 "Pointer Events"(`pointerdown`, `pointermove`, `pointerup`)라는, 손가락이든 마우스든 펜이든 한 가지 방식으로 다뤄주는 최신 웹 표준 이벤트를 썼다. 누르기 시작한 위치와 지금 위치의 차이(`deltaX`)를 계산해서 카드를 그만큼 옆으로 움직이고(`transform: translateX(...)`), 놓았을 때 그 차이가 기준값(80px)보다 크면 "읽음으로 표시"와 똑같은 함수를 실행한 뒤 카드가 옆으로 사라지는 애니메이션을 재생했다. 기준값보다 작으면 원래 자리로 되돌아간다.
  - 버튼과의 충돌 방지: 스와이프를 누르기 시작한 지점이 버튼 위라면(`e.target.closest('button')`) 스와이프를 아예 시작하지 않도록 했다. 처음에는 제목 링크 위에서도 스와이프를 막아뒀었는데, 이게 "컴퓨터에서만 되고 휴대폰에서는 안 되는" 버그의 원인이었다(아래 "주의할 점"에 자세히 적어둠). 지금은 제목 위에서도 스와이프가 시작되고, 대신 "많이 끌었는지"를 따로 기억해뒀다가(`wasDragged`) 진짜로 많이 끌었을 때만 그 뒤에 오는 클릭(=새 창으로 열기)을 취소한다.
  - 위아래 스크롤과 구분하기: 손가락이 화면을 위아래로 움직이는 건 "페이지 스크롤"이지 "카드 스와이프"가 아니다. 그래서 움직인 거리를 가로(`dx`)와 세로(`dy`)로 각각 재서, 세로로 더 많이 움직였으면 스와이프로 취급하지 않고 그냥 무시한다(페이지가 정상적으로 스크롤되게 놔둠).
  - 검증 방법: 실제 터치스크린이 없어서, `jsdom`으로 `pointerdown`→`pointermove`→`pointerup` 이벤트를 순서대로 코드에서 직접 발생시켜(마치 사람이 화면을 미는 것처럼) "많이 밀면 읽음 처리됨", "살짝만 밀면 처리 안 됨", "버튼 위에서 시작하면 스와이프 무시됨", "다른 학과 공지는 스와이프해도 아무 일 안 일어남", "제목 글자 위에서 시작해도 스와이프가 됨", "세로로 더 많이 움직이면 무시됨" 여섯 가지 경우를 전부 확인했다.
- 자바스크립트 `Set`이 "넣은 순서"를 기억한다는 성질을 날짜 대신 활용하기:
  - 왜 필요했는지: "읽은 공지는 최근에 읽은 순으로 보여달라"는 요청을 받았는데, "언제 읽었는지" 시각을 따로 저장해두지 않았었다. 새로 시각 정보를 추가하는 대신, 이미 쓰고 있던 `readSet`(읽은 공지의 링크를 모아두는 자바스크립트 `Set`)이 사실 "넣은 순서"를 그대로 기억하고 있다는 점을 활용했다(자바스크립트 `Set`은 나중에 넣은 게 뒤로 가는 순서를 보장한다).
  - 어디서 썼는지: `build_site.py`의 `refresh()` 함수에서, "읽은 공지" 탭을 볼 때만 `Array.from(readSet).reverse()`로 순서를 뒤집어서(가장 최근에 읽은 게 맨 앞에 오도록) 각 공지에 화면 표시 순서 번호를 매겼다.
  - CSS `order` 속성으로 "진짜 순서는 안 바꾸고 보이는 순서만 바꾸기": 공지 목록(`.notice-list`)이 이미 `display:flex`로 되어 있어서, 자바스크립트가 각 카드에 `order` 스타일 값만 넣어주면 HTML 안에서의 실제 위치(DOM 순서)는 그대로인 채로 화면에 보이는 순서만 바뀐다. 탭을 다른 걸로 바꾸면 `order` 값을 다시 비워서(`''`) 원래(날짜순) 순서로 돌아가게 했다.
  - 어떤 효과가 있었는지: 새로운 저장 값(읽은 시각)을 추가하지 않고도, 이미 있는 정보(넣은 순서)를 다른 각도로 활용해서 원하는 기능을 만들 수 있었다. "이미 가지고 있는 데이터에 내가 필요한 정보가 숨어있지 않은지" 먼저 살펴보는 습관이 코드를 더 간단하게 만들어준다.

## 다음에 비슷한 작업을 할 때 해야 할 점

- 새 사이트를 추가할 때는 `input/sites.csv`에 한 줄만 추가한다. 인천대의 다른 하위 사이트도 같은 게시판 템플릿(`board-table`)을 쓰면 parser 칸에 `inu_standard`라고만 적으면 바로 동작할 가능성이 높다.

- 사이트 구조가 다르면 `src/collect.py`에 새 파서 함수를 만들고 `PARSERS` 딕셔너리에 등록한다.
- 사이트 구조를 확인할 때는 `requests`로 페이지를 받아 파일로 저장한 뒤, 그 안에서 반복되는 태그 패턴을 찾는 방식이 편했다.
- 새 사이트를 추가할 때 `requests`로 받은 내용이 텅 비어 있거나 `<ik-root>`, `<app-root>` 같은 낯선 태그 하나만 있으면, 그 사이트는 자바스크립트로 화면을 그리는 곳일 가능성이 높다. 이럴 땐 바로 포기하지 말고, `playwright`를 임시로 설치해서 실제로 페이지를 열어보고 어떤 JSON 주소로 데이터를 받아오는지 엿들으면(`page.on('response', ...)`) 그 주소를 알아낼 수 있다. 찾고 나면 `playwright`는 지우고 `requests`만으로 매일 자동 수집하게 만들면 된다(무거운 도구는 "찾을 때만" 쓰고 "매일 도는 자동화"에는 안 남긴다).
- 필터링처럼 100% 정확할 수 없는 자동 판단 기능을 만들 때는, 걸러낸 결과를 숨기지 말고 "펼쳐서 확인 가능"하게 남겨두면 실수를 사람이 바로 발견하고 고칠 수 있다.
- "매일/매주 자동 업데이트되는 웹사이트"가 필요하면, 데이터 수집(collect)과 화면 생성(build)을 분리해두고, GitHub Actions + GitHub Pages 조합으로 무료로 자동화할 수 있다는 걸 기억해두면 좋다.
- GitHub 저장소 생성과 Pages 활성화는 브라우저에서 사람이 직접 클릭해야 하는 단계다(이 컴퓨터에는 `gh` CLI가 없어서 자동화 불가). 미리 이 단계가 필요하다는 걸 안내해두면 헷갈리지 않는다.
- "똑똑한 필터링 규칙"으로는 원하는 만큼 결과가 안 나올 때, 굳이 그 규칙을 계속 뜯어고치려 하지 말고 "이번엔 그냥 필터 없이 N페이지 전부 채워줘"처럼 사용자에게 직접 물어보고 그 사이트만 예외로 처리하는 것도 좋은 방법이다. `src/collect.py`에 이미 있는 부품들(`scrape_inu_js_link`, `with_page`, `classify_relevance`)을 그대로 가져다 쓰면 파이썬 한 줄짜리 임시 스크립트로도 쉽게 "N페이지 전부 채우기"를 할 수 있다 — collect.py 자체를 고칠 필요는 없다.

- 공개 GitHub 저장소로 배포하는 프로젝트는, 개인 메모가 섞인 메인 워크스페이스 저장소와는 분리해서 관리하는 게 안전하다. (이 프로젝트는 메인 저장소의 `.gitignore`에 등록해서 분리했다.)
- GitHub Actions cron 스케줄은 "예약한 시각 근처"에 실행되는 것이지 정확히 그 초까지 보장되지는 않는다. 몇 분 정도 늦게 실행돼도 정상이다.
- "이미 아는 글" 기준 증분 수집은 딱 한 가지 맹점이 있다: 사이트를 처음 추가할 때 놓친 예전 글은, 나중에 "업데이트를 한 번 더" 돌려도 자동으로 채워지지 않는다(이미 알고 있는 첫 글에서 바로 멈추기 때문). 예전 것까지 다시 채우고 싶으면 그 사이트의 저장 기록을 한 번 비우고 처음 추가하는 것처럼 다시 수집해야 한다.
- 새 사이트를 붙일 때 "글이 상단 고정이라 페이지마다 반복되는지"를 꼭 확인한다. 확인 안 하면 중복 저장되거나, 페이지 넘기기가 안 멈추는 문제가 생길 수 있다(실제로 데이터과학과 홈페이지에서 겪었다). 이번에 만든 "이미 담은 링크는 건너뛰기" 방식이 대부분의 경우를 막아주지만, 새 사이트를 추가할 때 한 번은 직접 결과를 눈으로 확인해보는 게 안전하다.
- "글이 며칠 안 됐는지"로 무언가를 걸러내는 규칙을 넣을 때는, 그 규칙이 "지금도 유효한 정보"까지 함께 걸러내진 않는지 의심해봐야 한다. 처음에는 "30일 넘은 글은 기간이 적혀 있어도 무시"하는 안전장치를 넣었는데, 실제로는 몇 달 전에 올려두고 학기 내내 유효한 공지(상단 고정 공지 등)를 이 규칙이 걸러내버리는 문제가 있었다. "오래된 글이니까 무시해도 되겠지"라는 가정은 실제 사이트를 보고 확인한 게 아니라 내 추측이었다는 게 문제였다 — 규칙을 넣을 때는 "왜 이 규칙이 필요한지"와 "이 규칙이 실수로 걸러낼 수 있는 진짜 사례가 있는지"를 같이 생각해야 한다.
- 날짜 범위를 정규표현식으로 읽어낼 때, 날짜 사이에 "(월)", "(금)" 같은 요일 표시가 끼어 있으면 정규표현식이 중간에서 끊길 수 있다("7/20(월) ~ 8/21(금)"에서 시작일 7/20을 놓치고 "~8/21"만 마감일로 잘못 읽는 식). 실제 사이트에서 나온 진짜 문장으로 테스트해보지 않았다면 발견하기 어려운 버그였다 — 정규표현식을 만들 때는 사람이 실제로 쓸 법한 여러 변형(요일, 괄호, 공백)을 미리 몇 개 적어두고 테스트하는 습관이 필요하다.
- localStorage/버튼 클릭처럼 브라우저에서만 확인되는 기능은, 코드만 읽는 검토로는 버그를 놓칠 수 있다(실제로 한 번 놓쳤었다). `node`와 `jsdom`으로 가짜 브라우저를 만들어 실제 클릭 이벤트를 발생시켜보는 방식이 훨씬 믿을 만하다. 다음에 비슷한 자바스크립트 기능을 추가할 때도 이 방식을 먼저 시도해볼 것.
- "기간 안에 있는 글만 포함"과 "마감 전인 글은 다 포함(아직 시작 안 했어도)"은 언뜻 비슷해 보여도 실제로는 다른 규칙이다. 사용자가 "지금 해당되거나 이후에 해당되는 것도"라고 말하면, "시작~마감 사이"가 아니라 "마감이 아직 안 지났는지"만 봐야 한다는 뜻일 수 있다 — 날짜 관련 요청은 "정확히 어떤 범위를 말하는지" 예시를 들어 확인하거나, 애매하면 더 넓게 잡아두고 사용자 반응을 보는 게 안전하다.
- 로그인이 필요한 비공개 서비스(이번엔 에브리타임)는, 사용자가 직접 아이디/비번을 주겠다고 해도 자동화하지 않는 게 맞다. 계정 소유자 본인이어도 "자동 접근"은 그 서비스의 이용약관을 어길 수 있고, 비밀번호를 공개 저장소가 있는 프로젝트 어딘가에 남기는 것도 위험하다. 이럴 땐 "왜 안 되는지"를 구체적으로 설명하고, 대신 할 수 있는 타협안(이번엔 바로가기 알림 카드)을 제안하는 게 좋다 — 무조건 안 된다고만 하기보다 대안을 같이 주면 사용자도 납득하기 쉽다.
- **jsdom 테스트를 "통과했다"고 해서 실제로 다 확인한 건 아닐 수 있다.** 스와이프 기능을 테스트할 때, 이벤트를 항상 카드 전체(`li`)에 직접 발생시켰는데, 실제 사용자는 카드 안의 특정 부분(이번엔 제목 글자)에서 손가락을 대고 시작한다. 테스트에서는 "어디를 눌러서 시작하는지"까지 실제 상황과 똑같이 흉내 내지 않으면, 테스트는 다 통과해도 진짜 사용 환경에서는 안 되는 버그가 남을 수 있다. 다음에 비슷한 터치/클릭 기능을 테스트할 때는 "사람이 실제로 어디를 누를지"까지 생각해서 그 지점을 이벤트 대상으로 삼아야 한다.
- **jsdom은 "로직"은 확인해줘도 "실제 화면에 그려지는지"는 확인해주지 못한다.** jsdom은 진짜 브라우저처럼 레이아웃을 계산하고 화면을 그리는(rendering) 기능이 없어서, "CSS 값이 맞게 설정됐는지"까지만 확인할 수 있고 "그 값이 실제 화면에 반영되는지"는 확인할 수 없다. 이번 "읽은 공지 정렬이 휴대폰에서 안 됨" 문제처럼, 코드 로직 자체는 맞는데 특정 브라우저가 화면을 다시 그리는 시점의 문제라면 jsdom 테스트로는 절대 못 잡아낸다 — 이런 종류의 버그는 실제 기기에서 눈으로 확인하는 것 외에는 검증할 방법이 마땅치 않다는 걸 인정하고, 사용자에게 "이 조치가 확실한 해결책인지는 실제 기기 확인이 필요하다"고 솔직하게 말해야 한다.
