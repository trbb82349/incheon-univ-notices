# 사용된 컴퓨터 지식/스킬 — 인천대 알림 모음

## 프로젝트 요약

인천대학교 여러 사이트의 공지를 자동으로 모아 보여주는 웹페이지를 만드는 중이다. 인천대 홈페이지 공지사항 게시판 1곳을 실제로 연동했고, 내 학과와 관련 없는 다른 학과 전용 공지를 자동으로 걸러서 접어두는 기능도 만들었다. `auto-update-site` 스킬을 적용해서 매일 아침 8시에 GitHub이 알아서 공지를 다시 확인하고 웹사이트를 갱신하도록 구조를 바꿨고(collect.py로 수집 → build_site.py로 웹페이지 생성 → GitHub Actions가 매일 자동 실행 → GitHub Pages로 공개), 실제로 GitHub 저장소(`incheon-univ-notices`)를 만들어 배포까지 완료했다. 이후 "오늘 날짜인 글만" 모으도록 개선하고, 하루에 게시판 1페이지(10건)보다 많이 올라와도 다음 페이지까지 자동으로 넘기며 확인하는 기능도 추가했다.

## 사용한 지식/스킬

- CSV 파일 (엑셀처럼 표 모양 데이터를 저장하는 아주 단순한 텍스트 파일):
  - 어디서 썼는지: `input/sites.csv`에 "사이트 이름, 주소, 분류"를 한 줄씩 적어두는 용도로 썼다.
  - 어떤 효과가 있었는지: 사이트 목록을 코드 안에 직접 적지 않고 파일로 따로 관리하니, 나중에 사이트를 추가하거나 뺄 때 코드를 건드리지 않고 이 파일만 고치면 된다.
- 웹 스크래핑 (사람이 웹사이트를 눈으로 보고 정보를 옮겨 적는 대신, 프로그램이 자동으로 웹페이지 내용을 읽어오는 기술) - requests + BeautifulSoup:
  - 어디서 썼는지: `src/collect.py`의 `scrape_inu_standard` 함수. `requests`로 게시판 페이지를 통째로 받아오고, `BeautifulSoup`로 그 안에서 "제목이 들어있는 자리", "작성 부서가 들어있는 자리" 같은 특정 부분만 골라냈다.
  - 어떤 효과가 있었는지: 사람이 매번 게시판에 들어가서 눈으로 확인할 필요 없이, 스크립트 실행 한 번으로 최신 공지 10개를 자동으로 가져올 수 있게 됐다.
  - 사이트 구조 확인 방법: 브라우저 개발자 도구 대신, `requests`로 페이지를 직접 받아서 파일로 저장한 뒤 그 안에서 반복되는 태그 패턴(여기서는 `<table class="board-table">` 안의 `<tr>` 하나가 공지 1개)을 찾았다.
- 정규표현식(regex)으로 패턴 찾기:
  - 어디서 썼는지: 공지 제목이나 작성 부서 이름에서 "OO학과", "OO학부", "OO전공"처럼 학과 이름 패턴을 자동으로 찾아내는 데 썼다 (`DEPT_PATTERN`).
  - 어떤 효과가 있었는지: "이 공지가 어느 학과 것인지"를 사람이 하나하나 읽지 않고, 이름 끝 글자 패턴만으로 자동 판단할 수 있게 됐다. 다만 완벽하지 않아서(예: 학과 이름이 아닌데 우연히 "학부"로 끝나는 조직명), 걸러진 공지도 숨기지 않고 펼쳐서 확인할 수 있게 안전장치를 뒀다.
- HTML의 `<details>`/`<summary>` 태그로 접이식 목록 만들기:
  - 어디서 썼는지: "다른 학과 공지"를 기본은 접혀 있지만 클릭하면 펼쳐지는 형태로 보여주는 데 썼다.
  - 어떤 효과가 있었는지: 자바스크립트 코드를 한 줄도 안 쓰고, HTML 태그만으로 클릭하면 펼쳐지는 목록을 만들 수 있었다.
- 데이터 수집과 화면 생성을 분리하기 (`data/data.json`을 가운데 두고 `collect.py` / `build_site.py`로 나눔):
  - 어디서 썼는지: `collect.py`는 "사이트에서 정보를 가져와서 파일(json)에 저장"만 담당하고, `build_site.py`는 "그 파일을 읽어서 웹페이지를 만드는" 역할만 담당하도록 나눴다.
  - 어떤 효과가 있었는지: 웹페이지 디자인만 바꾸고 싶을 때 다시 사이트를 스크래핑할 필요 없이 `build_site.py`만 다시 실행하면 된다. 두 역할이 섞여 있으면 나중에 하나를 고칠 때 다른 하나까지 실수로 건드릴 위험이 있다.
- GitHub Actions (사람이 컴퓨터를 켜두지 않아도, GitHub이 정해진 시간에 대신 코드를 실행해주는 무료 서비스):
  - 어디서 썼는지: `.github/workflows/update.yml`에 "매일 08:00 KST(한국시간)에 `collect.py`와 `build_site.py`를 순서대로 실행하고, 바뀐 내용이 있으면 자동으로 커밋·푸시"하도록 설정했다.
  - 어떤 효과가 있었는지: 내 컴퓨터를 매일 켜서 스크립트를 실행할 필요 없이, 자고 있어도 GitHub 서버가 대신 실행해준다.
  - cron 표기법 주의점: GitHub Actions의 cron 시간은 항상 UTC(세계 표준시) 기준이라서, 한국시간 08:00에 실행하려면 `0 23 * * *`(전날 23:00 UTC)처럼 9시간을 빼서 계산해야 한다.
- GitHub Pages (저장소 안의 `docs/` 폴더 내용을 무료로 인터넷에 공개해주는 서비스):
  - 어디서 썼는지: `docs/index.html`이 실제로 사람들이 접속해서 보는 주소(`https://아이디.github.io/저장소명`)가 된다.
  - 어떤 효과가 있었는지: 별도로 서버를 빌리거나 돈을 내지 않아도 웹사이트를 인터넷에 공개할 수 있다. 단, `docs/.nojekyll`이라는 빈 파일을 꼭 넣어야 정상 작동한다(안 넣으면 404 오류가 날 수 있음).
- 게시판 페이지 번호 자동으로 넘기기 (URL의 `?page=1`, `?page=2` 같은 부분을 코드로 바꿔가며 여러 페이지를 순서대로 확인):
  - 어디서 썼는지: `collect.py`의 `with_page` 함수가 주소의 페이지 번호만 바꿔서 다음 페이지 주소를 만들고, `collect_today_notices` 함수가 "오늘 날짜 글이 있는 동안 계속 다음 페이지로" 넘어가도록 했다.
  - 어떤 효과가 있었는지: 처음에는 게시판 1페이지(최신 10개)만 가져왔는데, 하루에 공지가 10개 넘게 올라오면 뒷부분을 놓치는 문제가 있었다. 페이지를 자동으로 넘기게 바꾸면서 "오늘 올라온 글은 몇 개든 다 가져오기"가 가능해졌다.
  - 언제 멈추는지: 게시판은 최신 글이 맨 위, 오래된 글이 아래에 오는 순서라서, 어떤 페이지에서 "오늘보다 오래된 날짜"인 글이 하나라도 나오면 그 뒤 페이지는 볼 필요가 없다고 판단하고 멈춘다. (무한정 넘기지 않도록 최대 15페이지라는 안전장치도 넣었다.)

## 다음에 비슷한 작업을 할 때 해야 할 점

- 새 사이트를 추가할 때는 `input/sites.csv`에 한 줄만 추가한다. 인천대의 다른 하위 사이트도 같은 게시판 템플릿(`board-table`)을 쓰면 parser 칸에 `inu_standard`라고만 적으면 바로 동작할 가능성이 높다.
- 사이트 구조가 다르면 `src/collect.py`에 새 파서 함수를 만들고 `PARSERS` 딕셔너리에 등록한다.
- 사이트 구조를 확인할 때는 `requests`로 페이지를 받아 파일로 저장한 뒤, 그 안에서 반복되는 태그 패턴을 찾는 방식이 편했다.
- 필터링처럼 100% 정확할 수 없는 자동 판단 기능을 만들 때는, 걸러낸 결과를 숨기지 말고 "펼쳐서 확인 가능"하게 남겨두면 실수를 사람이 바로 발견하고 고칠 수 있다.
- "매일/매주 자동 업데이트되는 웹사이트"가 필요하면, 데이터 수집(collect)과 화면 생성(build)을 분리해두고, GitHub Actions + GitHub Pages 조합으로 무료로 자동화할 수 있다는 걸 기억해두면 좋다.
- GitHub 저장소 생성과 Pages 활성화는 브라우저에서 사람이 직접 클릭해야 하는 단계다(이 컴퓨터에는 `gh` CLI가 없어서 자동화 불가). 미리 이 단계가 필요하다는 걸 안내해두면 헷갈리지 않는다.

## 다음에 비슷한 작업을 할 때 주의할 점

- 학교 사이트에 너무 빠르게 반복 요청을 보내면 부담을 줄 수 있으니, 사이트별 요청 사이에 약간의 딜레이(1초)를 두었다.
- 사이트 구조는 학교 쪽에서 언제든 바꿀 수 있어서, 스크래핑이 갑자기 안 되면 그 사이트의 HTML 구조가 바뀌었는지부터 확인한다.
- 정규표현식 기반 학과 이름 판단은 이름 끝 글자만 보기 때문에, "OO학부"처럼 학과가 아닌 조직명도 학과로 오인할 수 있다. 이런 경우는 `input/my_keywords.txt`를 조정하거나 예외 처리를 추가해야 한다.
- 공개 GitHub 저장소로 배포하는 프로젝트는, 개인 메모가 섞인 메인 워크스페이스 저장소와는 분리해서 관리하는 게 안전하다. (이 프로젝트는 메인 저장소의 `.gitignore`에 등록해서 분리했다.)
- GitHub Actions cron 스케줄은 "예약한 시각 근처"에 실행되는 것이지 정확히 그 초까지 보장되지는 않는다. 몇 분 정도 늦게 실행돼도 정상이다.
- "오늘 날짜 글만 모으기" 방식은 그날 새 글이 하나도 없으면(주말, 방학 등) 결과가 0건이 되는 게 정상이다. 스크립트가 고장 난 게 아니니 헷갈리지 않아야 한다.
